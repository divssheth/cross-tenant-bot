## Overview

This bot uses **multi-agent orchestration** to route Microsoft-related questions to specialized agents:

- **Triage agent** classifies user intent and hands off to the right specialist
- **Web agent** answers general Microsoft questions using search, docs, and acronym tools
- **License agent** answers M365 licensing questions using a deployed Foundry agent with a knowledge base

### Why Multi-Agent?

| Concern | Single Agent | Multi-Agent |
|---------|-------------|-------------|
| Tool isolation | All tools on one agent (confusing prompts) | Each specialist gets only its tools |
| Prompt size | One massive instruction set | Focused, concise instructions per agent |
| Foundry integration | Must proxy deployed agents behind wrapper tools | Native `AzureAIAgentClient` as a first-class participant |
| Evaluation | Can't measure routing accuracy | Routing introspection per-query |

## Architecture

### Data Flow

```
User Message (Teams)
        │
        ▼
┌──────────────────────────┐
│   FoundryAgentClient     │
│   foundry_agent_client.py│
│                          │
│  1. Lazy-init agents     │
│  2. Build Message list   │
│     (history + current)  │
│  3. Create workflow      │
│  4. workflow.run(msgs)   │
│  5. Extract response     │
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│   HandoffBuilder         │
│   Workflow                │
│   orchestrator.py        │
│                          │
│   Entry: triage          │
│   ┌─────────────┐       │
│   │   triage     │       │
│   │   (router)   │       │
│   └──┬──────┬────┘       │
│      │      │            │
│  ┌───▼──┐ ┌─▼─────────┐ │
│  │ web  │ │  license   │ │
│  │ agent│ │  agent     │ │
│  └──┬───┘ └──┬────────┘ │
│     │        │           │
│     └───┬────┘           │
│         ▼                │
│   AgentResponseUpdate    │
└──────────────────────────┘
        │
        ▼
Response → Teams
```

### Lifecycle

1. **Agents created once** — `_ensure_agents_created()` lazily initializes `AzureOpenAIResponsesClient` and all three agents on the first message. Agents are cached on `FoundryAgentClient`.
2. **Workflow created per message** — `_create_workflow()` builds a fresh `HandoffBuilder` workflow each time. This avoids stale internal state (accumulated tool calls and conversation history) that causes errors on subsequent turns.
3. **History injected** — `_build_chat_messages()` pulls prior turns from `ConversationStateManager` and converts them to `Message` objects (user/assistant roles). The current user message is appended last.
4. **Workflow runs** — `workflow.run(messages)` executes the handoff chain starting at triage.
5. **Response extracted** — Specialist responses come from `result.get_outputs()`. Triage direct responses (greetings, refusals) are captured via fallback event scanning.

## HandoffBuilder Workflow

The workflow is composed in `src/app/agents/orchestrator.py`:

```python
def create_workflow(triage, web_agent, license_agent=None):
    participants = [triage, web_agent]
    triage_targets = [web_agent]

    if license_agent:
        participants.append(license_agent)
        triage_targets.append(license_agent)

    builder = (
        HandoffBuilder(
            name="ms-expert-orchestration",
            participants=participants,
            termination_condition=_max_handoffs_termination(6),  # Safety net
        )
        .with_start_agent(triage)              # Entry point
        .add_handoff(triage, triage_targets)    # triage → specialists (one-way)
    )

    return builder.build()
```

**One-way routing:** Specialists do not hand back to triage. If an agent receives a misrouted question, it answers to the best of its ability or politely declines. This prevents handoff loops that can occur with bidirectional routing.

**Termination safety net:** `_max_handoffs_termination(6)` stops the workflow after 6 handoff steps to prevent infinite loops.

### API Breakdown

| Method | Purpose |
|--------|---------|
| `HandoffBuilder(name, participants)` | Creates a named workflow with a list of all participating agents |
| `.with_start_agent(agent)` | Sets the entry point (first agent to receive the message) |
| `.add_handoff(source, targets)` | Defines which agents `source` can hand off to (generates `handoff_to_<name>` tools) |
| `termination_condition=func` | Callback returning `True` to stop; used for `_max_handoffs_termination(6)` safety net |
| `.build()` | Returns a `Workflow` instance ready for `.run()` |

### Why One-Way Routing (No Back-Edges)

Earlier iterations allowed specialists to hand back to triage (`add_handoff(web_agent, [triage])`). This caused **handoff loops**:

1. Triage hands off to web_agent
2. Web agent answers but the framework continues processing
3. Web agent has nothing more to do, hands back to triage
4. Triage sees the same question, hands off to web_agent again
5. Repeat until termination

The fix: **one-way routing** (triage → specialists only) plus a `_max_handoffs_termination(6)` safety net. Specialists answer in-scope questions and politely decline out-of-scope ones.

### Why No `with_autonomous_mode()`

Autonomous mode adds a `"Continue assisting autonomously"` prompt after each agent response. This works well for multi-step tasks in local agents, but causes **infinite handoff loops** with Foundry-backed agents:

1. Triage hands off to license_agent
2. License agent answers the question
3. Autonomous mode sends "Continue assisting autonomously"
4. License agent has nothing more to do, hands back to triage
5. Triage sees the same question context, hands off to license_agent again
6. Repeat forever

The `HandoffBuilder`'s auto-generated handoff tools handle routing natively — no autonomous mode needed.

## Tool Types

The bot uses three distinct tool patterns:

### 1. `@tool` Decorator (Local Python Function)

```python
# _acronyms.py
from agent_framework import tool as ai_function

@ai_function
def decode_microsoft_acronym(acronym: str) -> str:
    """Decode a Microsoft or Azure acronym..."""
    # Looks up in MICROSOFT_ACRONYMS dict (40+ entries)
```

Registered on the agent via the `tools=` parameter in `client.as_agent()`.

### 2. Built-in Tool (Azure OpenAI Native)

```python
web_search = AzureOpenAIResponsesClient.get_web_search_tool()
```

A static method returning an Azure OpenAI Responses native web search tool. No local implementation needed.

### 3. MCP Tool (Model Context Protocol)

```python
ms_learn_mcp = MCPStreamableHTTPTool(
    name="microsoft_learn",
    url="https://learn.microsoft.com/api/mcp",
    description="Official Microsoft Learn MCP Server",
    approval_mode="never_require",
    request_timeout=60,
)
```

Connects to an external MCP server. The Microsoft Learn MCP exposes:
- `microsoft_docs_search` — Search official documentation
- `microsoft_docs_fetch` — Fetch full page content by URL
- `microsoft_code_sample_search` — Find official code samples

## Response Extraction

Response extraction happens in `FoundryAgentClient.chat()` (`foundry_agent_client.py`):

```python
result = await workflow.run(chat_messages)

# 1. Primary: specialist responses via handoff
outputs = result.get_outputs()
for output in outputs:
    if isinstance(output, AgentResponseUpdate) and output.text:
        response_text += str(output.text)

# 2. Fallback: triage direct responses (greetings, refusals)
if not response_text:
    for event in result:
        data = getattr(event, 'data', None)
        if data is not None and hasattr(data, 'text') and data.text:
            response_text = str(data.text)
```

**Why two paths?**
- When triage hands off to a specialist, the specialist's response appears in `get_outputs()` as `AgentResponseUpdate`
- When triage responds directly (greetings, out-of-scope refusals), there is no handoff output — the response is only available by scanning workflow events

## Adding a New Agent

### Step 1: Create the Agent Module

Create `src/app/agents/new_agent.py`:

```python
"""New specialist agent."""

from agent_framework import Agent
from agent_framework.azure import AzureOpenAIResponsesClient

NEW_AGENT_INSTRUCTIONS = """You are a specialist for [domain].
If the question is not about [domain], politely decline and explain your scope.
"""

def create_new_agent(client: AzureOpenAIResponsesClient) -> Agent:
    return client.as_agent(
        name="new_agent",
        instructions=NEW_AGENT_INSTRUCTIONS,
        description="Handles [domain] questions",
    )
```

### Step 2: Register in the Orchestrator

In `src/app/agents/orchestrator.py`:

```python
from app.agents.new_agent import create_new_agent

def create_agents(client, credential):
    triage = create_triage_agent(client)
    web_agent = create_web_agent(client)
    license_agent = create_license_agent(client, credential)
    new_agent = create_new_agent(client)                          # Add
    return triage, web_agent, license_agent, new_agent            # Update return

def create_workflow(triage, web_agent, license_agent=None, new_agent=None):
    participants = [triage, web_agent]
    triage_targets = [web_agent]

    if new_agent:                                                  # Add
        participants.append(new_agent)
        triage_targets.append(new_agent)

    # ... existing HandoffBuilder code ...

    # Note: routing is one-way (triage → specialists only).
    # No add_handoff(new_agent, [triage]) — specialists don't hand back.

    return builder.build()
```

### Step 3: Update Triage Instructions

In `triage_agent.py`, add a routing rule to `TRIAGE_INSTRUCTIONS`:

```
5. **[Domain] questions** → call handoff_to_new_agent
   - [Description of what this agent handles]
```

### Step 4: Add Test Data

In `src/app/eval/test_data.json`, add test cases:

```json
{
  "query": "A question for the new agent",
  "context": "",
  "ground_truth": "Expected answer summary",
  "category": "new_domain",
  "expected_behavior": "should_answer",
  "expected_agent": "new_agent"
}
```

### Step 5: Update FoundryAgentClient

In `foundry_agent_client.py`, update `_ensure_agents_created()` and `_create_workflow()` to pass the new agent through.
