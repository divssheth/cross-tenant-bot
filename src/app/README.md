# App Module

Multi-agent Teams bot using **Microsoft Agent Framework** for orchestration and **Microsoft 365 Agents SDK** for Teams hosting.

                         │   Microsoft Teams    │

                                  │
                         ┌────────▼─────────────┐
                         │  CrossTenantBot       │
                         │  (M365 Agents SDK)    │
                         └────────┬─────────────┘
                                  │
                         ┌────────▼─────────────┐
                         │  FoundryAgentClient   │
                         │  (lazy init, tracing) │
                         └────────┬─────────────┘
                                  │
                         ┌────────▼─────────────┐
                         │  HandoffBuilder       │
                         │  Workflow (per msg)   │

                                  │
                  ┌───────────────┼───────────────┐
                  │               │               │
          ┌───────▼──────┐ ┌─────▼──────┐ ┌──────▼───────┐

          │  (router)     │ │ (search +  │ │ (Foundry     │

## Agent Details

| Agent | Client Type | Purpose | Tools |
|-------|------------|---------|-------|
| **triage** | `AzureOpenAIResponsesClient.as_agent()` | Intent classification and routing | Handoff tools (auto-generated by `HandoffBuilder`) |
| **web_agent** | `AzureOpenAIResponsesClient.as_agent()` | General Microsoft questions | `decode_microsoft_acronym` (`@tool`), `web_search` (built-in), Microsoft Learn MCP |
| **license_agent** | `AzureAIProjectAgentProvider.get_agent()` | M365 licensing and subscriptions | Deployed Foundry agent's own tools + knowledge base |

## Key Design Decisions

- **Fresh workflow per message** — Agent instances are cached and reused, but `HandoffBuilder` workflows are rebuilt for each incoming message. This avoids stale tool-call state that causes "No tool output found" errors on subsequent turns.
- **No `with_autonomous_mode()`** — Autonomous mode sends "Continue assisting autonomously" prompts after each agent response. With Foundry-backed agents (license_agent), this causes infinite handoff loops because the agent answers, then has nothing else to do, triggering another cycle. Handoff tools handle routing natively without autonomous mode.
- **`name` not `agent_id`** — `AzureAIProjectAgentProvider.get_agent(name=...)` retrieves a persistent Foundry agent by its human-readable name (e.g. `unified-knowledge-agent-1`). No `asst_*` IDs are used.
- **Multi-turn context via `ConversationStateManager`** — Prior turns are converted to `Message` objects and prepended to each workflow run, giving the triage agent full conversational context.

## Key Components

### FoundryAgentClient (`agents/foundry_agent_client.py`)
Thin orchestration client managing the agent lifecycle:
- **Lazy initialization** — Creates `AzureOpenAIResponsesClient` and all agents on first message
- **Per-message workflow** — Calls `create_workflow()` for each message, runs `workflow.run(messages)`
- **Response extraction** — `result.get_outputs()` for specialist responses, fallback event scan for triage direct responses
- **Tracing** — Custom OpenTelemetry spans with agent/conversation/user attributes

### Triage Agent (`agents/triage_agent.py`)
Routes incoming messages to the correct specialist:
- Greetings/small talk → direct response (no handoff)
- Licensing questions → `handoff_to_license_agent`
- General Microsoft questions → `handoff_to_web_agent`
- Non-Microsoft topics → direct refusal

### Web Agent (`agents/web_agent.py`)
Handles general Microsoft questions with three tool types:
- **`@tool` decorator** — `decode_microsoft_acronym` for instant acronym decoding
- **Built-in tool** — `AzureOpenAIResponsesClient.get_web_search_tool()` for web search
- **MCP tool** — `MCPStreamableHTTPTool` connecting to `https://learn.microsoft.com/api/mcp`

### License Agent (`agents/license_agent.py`)
Retrieved from Foundry v2 via `AzureAIProjectAgentProvider.get_agent()`:
- Retrieves by `name` (e.g. `unified-knowledge-agent-1`) — no new `asst_*` instances created
- Uses the deployed agent's own instructions, tools, and knowledge base
- Provider must be kept alive for the agent's lifetime and closed on shutdown

### Graph RSC Client (`graph_rsc_client.py`)
Authenticates to Graph API using client credentials flow:
1. Retrieves client secret from Key Vault via UAMI
2. Gets token for target tenant
3. Calls Graph API with RSC permissions

### Conversation State (`conversation_state.py`)
Stores recent messages in-memory for context in 1:1 and group chats.
For channels, uses Graph API to retrieve message history.

## Usage

```bash
# Run the bot (from src/ directory)
python -m app

# Run evaluations (from src/ directory)
python -m app.eval.multi_agent_eval

# Run multi-turn evaluations only
python -m app.eval.multi_agent_eval --multi-turn-only

# Log results to Azure AI Foundry Portal
python -m app.eval.multi_agent_eval --log-to-foundry
```
